      ## Cu√°ndo utilizar OOP
      En Python TODO son objetos! Incluso las cadenas, los n√∫meros...
        dir(5.0)
      .
      Aunque Python sea un lenguaje OO, no nos "obliga" a usar objetos
      para nuestro c√≥digo, y de hecho es muy com√∫n no hacerlo.
      ¬øPor qu√©? Porque los namespaces, m√≥dulos, etc. (en general, la
      forma en que Python "estructura" todo) ya facilitan la "separaci√≥n"
      y encapsulaci√≥n de las diferentes capas de abstracci√≥n. Adem√°s,
      es habitual que se utilice el paradigma "stateless" (sin estado),
      es decir, que los objetos o entidades tengan una vida relativamente
      corta y que las funciones sean "puras" (no modifican lo recibido)

      Eso es especialmente cierto para algunos tipos de programas, como
      por ejemplo los que "producen" p√°ginas Web.
      Las pruebas, refactorizaci√≥n, y la concurrencia se facilitan tambi√©n.
      Por tanto, es bastante habitual que en ciencia de datos NO se use!
      .
      Entonces... ¬øCu√°ndo utilizar OOP?
      Suele ser √∫ltil e importante sobre todo en proyectos grandes.
      Tambi√©n cuando el estado s√≠ es "duradero" y/o complejo, por ejemplo
      juegos, o aplicaciones con GUI (de "escritorio", que no sea Web).
      Hay que valorar c√∫ando compensa o resulta necesario, y cu√°ndo no...
      .

      # Herencia (inheritance)
      Una clase que hereda de otra, tiene sus atributos y m√©todos,
      aunque puede modificarlos (sobreescr.) y/o a√±adir nuevos...
        class Car():
            ...
      .
        class ElectricCar(Car):
            def __init__(self, make, model, year, battery_type):

            ...
      Normalmente el m√©todo __init__ lo querremos sobreescr. siempre!
      Si no lo sobreescribimos (no aparece en el hijo) se queda "tal cual"

      ## super()
      super() se utiliza para aclarar (si hay ambig√ºedad) de qu√© clase es un
      m√©todo al que queremos llamar. Si hay herencia y sobreescribimos
      m√©todos, puede ser muy habitual (ej. en __init__). Sobre todo porque
      desde la clase hija se suele querer llamar (antes o despu√©s de hacer
      otras cosas) a lo equivalente que haga la clase padre en ese m√©todo...
      .
      Avanzado: con herencia m√∫ltiple puede hacer falta precisar m√°s, y
      para eso super() admite argumentos opcionales (...)

      ## Herencia m√∫tiple
      Muy avanzado: es cuando una clase hereda de dos o m√°s clases padre
      .
      La herencia m√∫ltiple es para usos muy concretos, no suele convenir,
      lo mismo que crear jerarqu√≠as (aunque sean con herencia simple) muy
      largas (muchos niveles de prof.) o muy complejas/amplias (...)
      .
      Utilizarla s√≥lo si es imprescindible y se entiende lo que implica!

      CONSEJO: casi siempre se puede lograr lo mismo (o muy equivalente)
      que con la herencia (tanto simple, como sobre todo si es m√∫ltiple), con
      composici√≥n (una clase contiene atributos cuyo tipo son otras clases).
      A veces no queda tan compacto el c√≥digo fuente, pero suele ser mucho
      m√°s f√°cil de mantener (hacer cambios a futuro), entender, y probar...!
      Herencia: A es un B
      Composici√≥n: A tiene un B.

      # Decoradores (decorators)
      Avanzado:
      "The decorator processes the decorated function in some way and returns it

      .
      Son un caso de de funciones de orden superior (que devuelven otras funciones),
      podemos verlos como funciones que "envuelven" o enmascaran / controlan a otras,
      por ejemplo para hacer algo justo antes y/o despu√©s de que se ejecuten.

      .

                ...

                print(f"function will be executed now")
                # invocar ahora a f() "normalmente" y guardar su resultado!
                result = f(*args, **kwargs)
                print(f"function running time:", time.time() - start)

            return _timer  # devolver la nueva funci√≥n "decorada"
      .
        @my_timer  # con esta notaci√≥n, Python sustituye autom. f por la decorada
        def f(x): return x * 2

      ## Bloque try / except
      Ejemplo muy completo:

            s = f.readline()
            i = int(s.strip())
        # para capturar un tipo de excepci√≥n concreto

        # lo mismo pero sin recoger/usar la info del error
        except ValueError:

        # para capturar cualquier excepci√≥n (tras las concretas). NO RECOMENDABLE!

            print("Finished OK")

        # parte opcional (siempre la √∫ltima) que se ejecuta EN CUALQUIER CASO (al fina)

      # with y "cleanup"
      Semi-avanzado: complicado de entender "a fondo" y sobre todo en detalle
      c√≥mo funciona internamente (o hacer nosotros objetos / funciones / etc.
      adecuados para ser utilizado por terceros mediante with), pero s√≠ que es
      muy habitual y recomendable usarlo en algunos casos.
      Con ficheros por ejemplo, es pr√°ctica habitual / casi imprescindible!
      La motivaci√≥n es que "gestiona" objetos que se destruyen/"limpian" autom.:
      : https://docs.python.org/3/tutorial/errors.html#predefined-clean-up-actions
      Ejemplo:

                print(line, end="")

      ## ¬øPor qu√© hace falta?

            f.write('hello, world!')
      .
      Porque en realidad equivale a algo (aprox.) como esto:
        f = open('hello.txt', 'w')
        try:
            f.write('hello, world')
        finally:  # muy importante!
            f.close()
      .
      Es decir, nos facilita la gesti√≥n de errores, que deber√≠amos
      hacer siempre, especialmente al trabajar con ficheros!
      M√°s detalles (avanzado) en la doc. oficial.

      # Aserciones (assert)
      Avanzado: "Mathematical expressions showing the current

      "The overall point is that if something does go wrong, we want to make it
      completely obvious as soon as possible. It's easier to catch incorrect data

      a complement"

      Es decir, no usar s√≥lo aserciones sin tests de unidad (o sistema)!
